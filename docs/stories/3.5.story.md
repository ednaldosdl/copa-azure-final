## Story

**Como** desenvolvedor do squad do Living Lab (papel `@dev`, sob o gate `@architect`),
**eu quero** provisionar *just-in-time* um registro `users` (com `entra_oid` vinculado) no primeiro login/checkout de um cliente **nato-CIAM** (que nunca teve conta v1), expondo um endpoint `GET /api/v2/me` — restrito ao esquema **CIAM** (`CiamOnly`, nunca ao esquema Admin/workforce) — que resolve `X-Entra-OID` → `users.id` por precedência **oid → email-link → insert** (ADE-007 v1.2 Invariante 8),
**para que** um cliente que se cadastrou diretamente no Microsoft Entra External ID (sem nunca ter existido no v1) consiga completar uma compra v2 e se torne **cidadão de primeira classe** na base `users` (hoje invisível ao dashboard admin, que já lê `users`) — hoje `PurchaseRequest.UserId` (int, obrigatório) exige uma linha `users` pré-existente, gap **não** coberto pela migração batch da ADE-007 Inv 6 — e **sem que um token de admin possa jamais criar ou vincular essa linha em nome do operador** (Invariante 8.2).

3. **AC-3 — Fence de esquema `CiamOnly`: `/api/v2/me` autorizado SOMENTE para o esquema CIAM, nunca para o Admin (workforce).** Verificado na fonte (`Gateway/Program.cs`): o transform que injeta `X-Entra-OID` é **global** (linhas 94-116 — roda para todos os clusters e qualquer usuário autenticado, seja `CiamScheme` ou `AdminScheme`, definidos nas linhas 236-237), e a `DefaultPolicy` das rotas v2 aceita **CIAM OU Admin** autenticado (linhas 393-396, `new AuthorizationPolicyBuilder(CiamScheme, AdminScheme).RequireAuthenticatedUser()`). **Sem este fence**, um token de **admin** (workforce) — que também carrega o claim `oid` — satisfaria `/api/v2/me`, o gateway injetaria o `oid` do **operador** como `X-Entra-OID`, e o resolve-or-provision (AC-2) criaria/vincularia uma linha de **CLIENTE** com a identidade do admin — corrompendo a base de clientes (e, pelo arm de LINK por email, potencialmente vinculando a linha de um cliente real ao operador). A nova policy `CiamOnly` é o **espelho exato** de `AdminOnlyPolicy` (já existente, linhas 388-389, 408-410 — `RequireRole(AdminRole)` sobre o esquema Admin), exigindo o esquema **`Ciam`** em vez do `Admin`. Um token workforce roteado pelo selector ao esquema `Admin` **não** satisfaz `CiamOnly` → **401/403** (forma exata — fixar o esquema vs. só checar autenticação — é decisão de `@dev`, mesma reflexão já registrada para `AdminOnlyPolicy`) — o resolve-or-provision **nunca** corre com um `oid` de admin.
   *(Fonte: ADE-007 v1.3 Invariante 8.2 — resolvida integralmente nesta ADE, nova nesta atualização da story; verificado `Gateway/Program.cs` linhas 94-116 (transform global de `X-Entra-OID`), 236-238 (`CiamScheme`/`AdminScheme`/`SelectorScheme`), 393-396 (`DefaultPolicy`), 388-389/408-410 (`AdminOnlyPolicy`, o espelho a seguir).)*

5. **AC-5 — `users.password NOT NULL` sem `DEFAULT` resolvido por SENTINELA fail-closed (RESOLVIDO por ADE-007 v1.2 Invariante 8.1 — não mais condicional):** verificado em `fifa2026-api/database/schema.sql` linha 20 (`password NVARCHAR(255) NOT NULL`).
